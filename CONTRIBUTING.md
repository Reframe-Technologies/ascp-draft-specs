# üõ† Contribution Process

We use a **Developer Certificate of Origin (DCO)** workflow.  
No separate CLA is required at this stage.

To contribute:

1. **Fork the repository**
2. **Create a feature branch**
3. Make your edits or additions
4. **Sign your commit** using the DCO format  
   (`git commit -s -m "‚Ä¶"` adds the required signature)
5. Submit a Pull Request
6. Participate in discussion and revision if needed

Every commit must contain the `Signed-off-by:` line that states you have the right to contribute the code/text under the project licenses.

# ‚öñÔ∏è Licensing Expectations

Contributions are accepted under the licensing model defined in the repository‚Äôs `LICENSE` file:

### **Specification Text**
- Accepted under terms consistent with the **IETF Trust Legal Provisions (TLP)**  
- Derivative works MAY NOT present themselves as the original specification  
- Text contributions may later be submitted to the **IETF**  
- Upon IETF submission, contributions become governed by the IETF Trust

### **Code Components in Specs (ABNF, schemas, examples)**
- MUST be contributed under **BSD-2-Clause**

### **Reference Implementations**
- MUST be contributed under **Apache 2.0**

By submitting contributions, you agree to these terms.

# üåê Path Toward IETF Standardization

ASCP is explicitly being developed with the intention of:

- Producing **Internet-Drafts (I-Ds)**  
- Participating in a **Birds-of-a-Feather (BoF)** discussion  
- Moving toward a **Working Group (WG)**  
- Publishing **Standard-Track RFCs**

As such, contributions should align with IETF norms:

- Clear, technical, non-promotional writing  
- Protocol behaviors described as **MUST/SHOULD/MAY** (RFC 2119 / 8174)  
- Precise definitions and unambiguous semantics  
- Security and privacy considerations included early
